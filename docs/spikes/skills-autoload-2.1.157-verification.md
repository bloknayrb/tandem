## What 2.1.157 changed

CHANGELOG (2.1.157): *"Plugins in `.claude/skills` directories are now automatically loaded, no marketplace required."*

Per the skills reference, this is the now-stable behavior:

- A skill is `<dir>/SKILL.md`. The **command name comes from the directory name**, not the `name:` frontmatter field. For `.claude/skills/<dir>/SKILL.md`, `name:` is only the display label shown in listings.
- **No explicit registration in `settings.json` is needed.** Project skills auto-load from `.claude/skills/` in the start dir and every parent up to the repo root.
- Custom commands (`.claude/commands/*.md`) have been merged into the same model: a command and a skill with the same name both create `/name`; if both exist, **the skill takes precedence**.
- Skill levels: enterprise > personal (`~/.claude/skills/`) > project (`.claude/skills/`). **Plugin skills use a `plugin-name:skill-name` namespace, so they cannot collide as commands with the other levels.**
- A skill folder becomes a *plugin* (`<name>@skills-dir`) **only if** it contains a `.claude-plugin/plugin.json`. Otherwise it is a plain skill.

Reconciling the worktree against CLAUDE.md: this is the `master` checkout. CLAUDE.md's autoskills summary references a larger set (issue-pipeline, accessibility, vitest, tauri-v2, etc.) that is **not present on master** — those live on other branches / are generated by `autoskills`. For people *working on Tandem*. Five skills, each `disable-model-invocation: true` (manual `/`-invoke only):

| Directory | `name:` | Invokes as | `disable-model-invocation` |
|---|---|---|---|
| `.claude/skills/changelog/` | `changelog` | `/changelog` | true |
| `.claude/skills/dev-server/` | `dev-server` | `/dev-server` | true |

In every case the `name:` field equals the directory name, so the invocation command is unambiguous and unchanged by 2.1.157. The `disable-model-invocation: true` on all five means they never auto-trigger — they only appear when typed — which is correct for these (they kill dev servers, run E2E, etc.).

### Surface B — the distributable Tandem plugin (`.claude-plugin/` + `skills/tandem/`)

This is Tandem packaged as a Claude Code plugin that *end users* install. It is **separate** from `.claude/skills/`:

- `.claude-plugin/plugin.json` — plugin `tandem` v0.8.0, declares `mcpServers` (`tandem`, `tandem-channel`) + an `experimental.monitors[]` entry.
- `.claude-plugin/marketplace.json` — marketplace `tandem-editor` listing the `tandem` plugin from `github:bloknayrb/tandem`.
- `skills/tandem/SKILL.md` — the single source-of-truth skill body (`version: 3`), used by both the marketplace plugin AND `tandem setup`'s `installSkill()` (which writes a copy to `~/.claude/skills/tandem/SKILL.md` from `src/cli/skill-content.ts`'s `SKILL_CONTENT`).

## Verification against the tasks in #1044

1. **Each shipped skill loads and is invokable.** ✅ (analysis) — All five `.claude/skills/*` directories have valid frontmatter; directory names are valid command names (lowercase, hyphenated); `name:` matches the directory. They invoke as `/changelog`, `/dev-server`, `/e2e`, `/e2e-debug`, `/screenshots`. Note: the CLAUDE.md-referenced `/issue-pipeline` does **not** exist on master (no `issue-pipeline` dir present) — that is a branch/autoskills artifact, not a regression.

2. **None double-registered or odd-named.** ✅ with one documented edge case:
   - **Within `.claude/skills/`:** no double-registration. No skill folder carries a `.claude-plugin/plugin.json`, so none loads twice (once as skill, once as `@skills-dir` plugin).
   - **`settings.json`:** the only `enabledPlugins` entry is `dev-tools@claudestuff-marketplace` (an unrelated external plugin). No Tandem skill is registered there. The `claude plugin …` Bash permission allowances in `settings.json` are permission grants, not registrations.
   - **`/diverge`:** exists only as `.claude/commands/diverge.md` (no same-named skill) — no command/skill precedence conflict.
   - **Edge case (benign, pre-existing):** a user who installs the **marketplace plugin** *and* runs **`tandem setup`** gets the `tandem` skill twice: once as the plugin skill `tandem:tandem` (namespaced; `userFacingName="tandem"`, per `marketplace-install-spike.md` F2.2) and once as the personal skill `/tandem` (`~/.claude/skills/tandem/SKILL.md`). These **do not collide as commands** (the plugin form is namespaced), but the model-invocation listing would show two entries with identical `tandem` descriptions. This duplication is **not introduced by 2.1.157** — personal `~/.claude/skills/` has always auto-loaded; 2.1.157 only added auto-load for *project* `.claude/skills/`, which Surface B does not use. No action taken; see "Recommendation" below.

3. **Generic auto-trigger skills still trigger.** N/A on master — the generic auto-trigger skills named in CLAUDE.md (accessibility, vitest, tauri-v2, etc.) are not checked into `.claude/skills/` on this branch, so there is nothing to regress here. The five present skills are all `disable-model-invocation: true` by design and intentionally do *not* auto-trigger.

4. **Behavior changed → document the fix.** No behavior change affecting Tandem's shipped skills. 2.1.157's auto-load is exactly the discovery mechanism these skills already relied on; the directory-name-derived command names and `disable-model-invocation` flags remain honored. **No restructuring of `.claude/skills/` is required.**

## Why nothing is broken by the "plugins are auto-loaded" wording

The 2.1.157 note says skills in `.claude/skills` are loaded *"no marketplace required"* — i.e., a skill folder no longer needs to be published/installed through a marketplace to be discovered. It does **not** mean every `.claude/skills/*` folder is reinterpreted as a plugin. A folder is only treated as a plugin when it contains `.claude-plugin/plugin.json` (loading as `<name>@skills-dir`). None of Tandem's five contributor skills do, so none gains MCP-server/hook/agent plugin semantics or a namespaced command.
