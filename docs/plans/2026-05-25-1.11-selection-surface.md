## Design — the load-bearing decisions

### D1. The full format set in the popup: reuse, don't duplicate

The popup must gain Strike, Code, Link, headings, lists, blockquote, code block — today only on `FormattingToolbar`. **Do not re-implement them in `Toolbar.svelte`.** Reuse `FormattingToolbar` inside the popup's format pill via a variant prop:

- Add a `variant?: "bar" | "popup"` (or `showHistory?: boolean`) prop to `FormattingToolbar`. In `"popup"` mode it **omits Undo/Redo** (those stay on the bar + `Ctrl+Z/Y`; the mockup's format pill has no history buttons) and adopts the popup's compact spacing.
- The popup renders `<FormattingToolbar variant="popup" {editor} />` as the top pill, then the annotate pill (highlight swatches + Annotate) below it.
- **Link inline-input in the popup:** `FormattingToolbar`'s link affordance is an inline-input popover (`toolbar-link-input`). It must work inside the popup's `position:fixed`, flipping container. Verify it anchors to the link button (not the viewport) and doesn't get clipped; if it currently assumes the bar's layout, constrain it to render within the popup. This is the **highest-risk integration point** — call it out for the svelte/crdt reviewers and manual testing.

> **Alternative considered:** extract a shared `FormattingControls.svelte` (buttons minus history) consumed by both `FormattingToolbar` and the popup. Cleaner separation but more churn (new file + rewire the bar). **Recommendation: the variant prop** — less surface area, keeps one source of truth for the button set. Flag for Bryan/review; switch to extraction if the variant prop bloats `FormattingToolbar` with layout branches.

> **Svelte note (plan-review LOW):** any variant-derived value inside `FormattingToolbar` (the omit-history flag, compact-spacing branch) must be `$derived(variant === "popup" ? … : …)`, never a plain `const`, so it tracks if `variant` ever becomes dynamic.

**Redundancy is intentional:** when the bar is visible, formatting appears both on the bar and in the popup. Bryan's explicit decision ("make the popup always show full, even when the formatting bar is there"). Document in the PR body so it doesn't read as a bug.

- **Mirror the Decorations control too (2026-05-26).** The popup is the full mirror of the formatting bar, and 1.13 put the Decorations split button in the bar — so the popup must also expose Decorations (eye + caret), reusing `DecorationsMenu.svelte`. This is what lets the bar be hidden without stranding the decoration/authorship toggles (it's the reachability guarantee that replaced the old title-bar home). Thread the same decoration props + `onUpdateDecorations` + `onOpenSettings` from `App.svelte` into `Toolbar.svelte`. Mount it outside any `overflow:hidden` track (same clip-escape reasoning as the bar). Reuse `DecorationsMenu` as-is — do not fork it.

### D2. Annotate popover — modifier-gated submits, Enter is a newline, anchored, no quoted-text echo

The keybindings (Conflict #5 **as overridden by Bryan 2026-05-26** — see `conflicts-resolved.md` → Applied Overrides → "Sub-PR 1.11 annotate popover keybindings"): **`Enter` (no modifier) = newline inside the textarea**, **`Alt+Enter` = Note to self**, **`Ctrl+Enter`/`⌘Enter` = Send to Claude**. Today plain Enter = comment (send to Claude). This is a deliberate behavior change, not a refactor:

- Rewrite `handleTextareaKeyDown` (L395-403): `Alt+Enter` → `submitAsNote()`; `Cmd+Enter`/`Ctrl+Enter` → `submitAsComment()`; **plain `Enter` and `Shift+Enter` → newline — do NOT `preventDefault()`, let the `<textarea>` insert the line break** (today the handler intercepts plain Enter to submit; that intercept must be removed); `Escape` → dismiss. Each submit branch calls `e.preventDefault()` only on its own modifier combo, never on bare Enter.
- **Add an empty-content guard to BOTH submit paths (plan-review HIGH-convergence — flagged by BOTH crdt-reviewer and annotation-model-reviewer).** Today `submitAsNote()` (L390-391) has **no** empty-text guard (it was button-only) whereas `submitAsComment()` early-returns on empty (L385). Now that a key combo can fire `submitAsNote()`, add `if (!annotationTextTrimmed) return;` to it (mirroring `submitAsComment()`), and disable **both** the "Note to self" and "Send to Claude" buttons when empty (`disabled={!annotationTextTrimmed}`). An empty `Alt+Enter` or `Ctrl/Cmd+Enter` must be a no-op — create nothing in the Y.Map.
- Reorder/label the buttons: **Note to self** (`⌥⏎` hint) and **Send to Claude** (`⌘⏎` hint). Keep testids `popup-note-submit` / `popup-comment-submit` verbatim (the manifest contract) even though Send-to-Claude is now labeled "Send to Claude" rather than "Comment" — update the visible label, keep the testid.
- **The old "missed-send risk (MED)" is now resolved, not mitigated.** The earlier plan (when Enter = note) carried a real reverse-failure mode: a user with send-muscle-memory hits Enter and silently gets a private note Claude never receives. Under this scheme **plain Enter submits nothing** — it just inserts a line break — so an accidental Enter costs nothing and the failure mode is gone. Two affordances still ship (now as plain good practice, not risk mitigation): (a) both buttons render their key hint (`⌥⏎ Note to self`, `⌘⏎ Send to Claude`) so the mapping is discoverable; (b) post-action feedback distinguishes "Saved as private note" vs "Sent to Claude".
- Anchored popover that flips above/below the selection; **no quoted-text echo** of the selected text (the mockup omits it). Largely restyling the existing annotate-mode block + the keybinding change.
- **Tests that assert plain-Enter=comment must change to plain-Enter=newline-no-submit**, and add `Alt+Enter`=note + `Ctrl/Cmd+Enter`=comment cases. This is the test-update signal the per-spec assertions will catch — fix the behavior/labels, and update the assertions to the new contract; do **not** silence them.

### D3. Optional bar + swap control — mirror the panel-visibility idiom

- Add `formattingBarVisible: boolean` (default `true`) to settings, following the `leftPanelVisible` idiom exactly: interface field, `DEFAULTS`, `normalizeKnownFields` (`=== false ? false : DEFAULTS`), schema bump + pure migration step (v9→v10 — **note this stacks on top of 1.13's v8→v9**; if 1.13 hasn't bumped yet at implementation time, reconcile to whatever `CURRENT_SCHEMA_VERSION` is on the branch). Add nothing to `REMOVED_FIELDS`.
- Wrap the `<FormattingBar>` invocation in `{#if settingsState.settings.formattingBarVisible}` (the bar has no internal visibility gate today, so a direct `{#if}` is correct — no LayoutModel wrapper needed since there's no multi-factor override). State-loss on unmount (an in-flight link URL, an open heading menu) is real but harmless and consistent with how the codebase already gates TitleBar children with `{#if}` — the bar holds no persistence-worthy local state (unlike the editor/panels, which use display-toggle). Note it in the PR body; no display-toggle conversion needed (plan-review LOW).
- **Swap/hide affordance:** the bar gains a trailing "hide bar" control (`formatbar-hide-btn`) that sets `formattingBarVisible:false`. Because the bar is gone once hidden, **restoring** uses non-bar affordances:
  - A command-palette action `Toggle formatting bar` (register in `builtin.svelte.ts` + App.svelte handler calling `updateSettings({ formattingBarVisible: ! })`), and
  - An Appearance settings toggle (`appearance-formatting-bar`).
  - A keyboard shortcut is **optional** for 1.0 — propose `Ctrl+Alt+F` only if it doesn't collide (grep first); otherwise palette + settings suffice. Decide during implementation; don't invent a colliding binding.
- When the bar is hidden, formatting is fully reachable via the always-full popup (D1) — that's the whole point of making the popup full first.

## Plan-review addenda (2026-05-26, three agents)

The revised plan was re-reviewed by `svelte-migration-reviewer`, `crdt-reviewer`, and `annotation-model-reviewer`. No HIGH findings; the core is sound. Four items to implement against:

1. **(svelte MED) Escape must not bubble out of the popup's link input.** The popup mounts a window-level Escape listener that calls `dismissPopup()` unconditionally (`Toolbar.svelte` L274-285). `FormattingToolbar`'s `handleLinkInputKeyDown` (L146-154) calls `preventDefault()` on Escape but **no `stopPropagation()`** — so once `FormattingToolbar` renders inside the popup, Escape while typing a URL would dismiss the link input AND tear down the whole popup. **Fix:** add `e.stopPropagation()` to the Escape (and Enter) branch of `handleLinkInputKeyDown`. Harmless in the bar context (no window Escape listener there). The `clickOutside` path is fine — it uses non-portaled `node.contains()`.
2. **(crdt MED) Selection-preservation pattern must hold for every reused popup control.** A popup button bound via `onclick` (not `onMouseDown`+`preventDefault`) risks disturbing the editor selection. **Audit result: `FormattingToolbar` already uses `onMouseDown={withPreventDefault(...)}` on every button** (incl. heading/list/link/HR/codeblock) — so the `variant="popup"` reuse needs **no** per-button event-mode change. The one control to verify is the **mirrored `DecorationsMenu`**: its trigger/rows must use the same mousedown+preventDefault discipline in the popup mount so interacting with it can't collapse the selection / dismiss the popup before a follow-up Annotate. (Note: the popup dismisses on selection-collapse / Escape / outside-scroll — not on plain editor blur — so this is belt-and-braces, but keep it consistent.)
3. **(annotation LOW) Make the empty-content invariant structural.** Add a guard inside `createAnnotation` itself — `if (!content.trim() && type !== "highlight") return;` — so the invariant lives at the write seam, not only in `submitAsNote`/`submitAsComment`. Defense-in-depth against a future caller. Keep the D2 textarea guards too.
4. **(annotation impl-note) Key-branch order.** In `handleTextareaKeyDown`, test `e.altKey` (note) and `e.metaKey/e.ctrlKey` (comment) on their own branches; **plain Enter must hit no submit branch at all** (just let the textarea insert the newline — do not `preventDefault`). Never let a note-intent keystroke fall through to a comment submit.

**New tests these add (beyond the D2 test list):** (crdt LOW) a structural-command-then-submit case — type in the textarea → click H2 → Send to Claude → assert the resulting Y.Map `range` matches the heading-prefix-shifted content offsets (proves prefix-accounting survives format-in-popup); and the seam-level audience-tag assertion (Alt+Enter→note/private, Ctrl/Cmd+Enter→comment/outbound, observer emits the comment not the note) is **non-negotiable** — it's the only test that catches `submitAsNote` being accidentally wired to `createAnnotation("comment")`.
