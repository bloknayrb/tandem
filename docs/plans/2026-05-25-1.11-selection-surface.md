# Sub-PR 1.11 — Selection surface + optional formatting bar

## Context

Umbrella `feat/design-system-impl`. This is the **second** of the two PRs resolving the updated **Conflict #5** (`conflicts-resolved.md`, updated 2026-05-25). It ships **after** the Decorations control PR (1.13), which gives the authorship toggle its title-bar home — so this PR can safely remove authorship from the formatting bar and make the bar hideable without any interval where authorship lacks an on-screen affordance.

**What it delivers:**
1. The on-text selection popup (A8, ADR-027) becomes **always the full stacked surface** — a format pill (B / I / S / code / link · headings / lists / quote / codeblock) over an annotate pill (highlight swatches + an **Annotate** button → anchored note popover).
2. The annotate popover uses the locked keybindings: **Note to self = `⏎`**, **Send to Claude = `⌘⏎` / `Ctrl+Enter`** (this **inverts** today's Enter semantics — see D2).
3. The persistent floating **formatting bar becomes optional/hideable** via a swap control, governed by a new `formattingBarVisible` setting (default `true`).
4. The authorship toggle is **removed** from the formatting bar (its home is now the title-bar Decorations control from 1.13).

**Display/UI only; no `src/server/` changes; ADR-027's audience-first model stays structurally primary.**

> **Dependency:** this PR assumes `DecorationsMenu.svelte` (the title-bar authorship home) already shipped in 1.13. Ship order enforces it.

## Design — the load-bearing decisions

### D1. The full format set in the popup: reuse, don't duplicate

The popup must gain Strike, Code, Link, headings, lists, blockquote, code block — today only on `FormattingToolbar`. **Do not re-implement them in `Toolbar.svelte`.** Reuse `FormattingToolbar` inside the popup's format pill via a variant prop:

- Add a `variant?: "bar" | "popup"` (or `showHistory?: boolean`) prop to `FormattingToolbar`. In `"popup"` mode it **omits Undo/Redo** (those stay on the bar + `Ctrl+Z/Y`; the mockup's format pill has no history buttons) and adopts the popup's compact spacing.
- The popup renders `<FormattingToolbar variant="popup" {editor} />` as the top pill, then the annotate pill (highlight swatches + Annotate) below it.
- **Link inline-input in the popup:** `FormattingToolbar`'s link affordance is an inline-input popover (`toolbar-link-input`). It must work inside the popup's `position:fixed`, flipping container. Verify it anchors to the link button (not the viewport) and doesn't get clipped; if it currently assumes the bar's layout, constrain it to render within the popup. This is the **highest-risk integration point** — call it out for the svelte/crdt reviewers and manual testing.

> **Alternative considered:** extract a shared `FormattingControls.svelte` (buttons minus history) consumed by both `FormattingToolbar` and the popup. Cleaner separation but more churn (new file + rewire the bar). **Recommendation: the variant prop** — less surface area, keeps one source of truth for the button set. Flag for Bryan/review; switch to extraction if the variant prop bloats `FormattingToolbar` with layout branches.

> **Svelte note (plan-review LOW):** any variant-derived value inside `FormattingToolbar` (the omit-history flag, compact-spacing branch) must be `$derived(variant === "popup" ? … : …)`, never a plain `const`, so it tracks if `variant` ever becomes dynamic.

**Redundancy is intentional:** when the bar is visible, formatting appears both on the bar and in the popup. Bryan's explicit decision ("make the popup always show full, even when the formatting bar is there"). Document in the PR body so it doesn't read as a bug.

### D2. Annotate popover — invert Enter, anchored, no quoted-text echo

The locked keybindings (Conflict #5) are **Note to self = `⏎`**, **Send to Claude = `⌘⏎`/`Ctrl+Enter`**. Today Enter = comment (send to Claude). **This is a deliberate behavior inversion**, not a refactor:

- Rewrite `handleTextareaKeyDown` (L395-403): `Enter` (no shift, no meta/ctrl) → `submitAsNote()`; `Cmd+Enter`/`Ctrl+Enter` → `submitAsComment()`; `Shift+Enter` → newline (default); `Escape` → dismiss.
- **Add an empty-content guard to `submitAsNote()` (plan-review HIGH-convergence — flagged by BOTH crdt-reviewer and annotation-model-reviewer).** Today `submitAsNote()` (L390-391) has **no** empty-text guard (it was button-only; the button isn't disabled), whereas `submitAsComment()` early-returns on empty (L385). Once Enter maps to `submitAsNote()`, a reflexive Enter on an empty textarea persists a zero-content private `note` into the Y.Map. Add `if (!annotationTextTrimmed) return;` to `submitAsNote()`, and disable the now-primary "Note to self" button when empty (mirror the existing `disabled={!annotationTextTrimmed}` on the comment button, L499). Empty-Enter must be a no-op.
- Reorder/re-emphasize the buttons: **Note to self** primary (`⏎` hint), **Send to Claude** secondary (`⌘⏎` hint). Keep testids `popup-note-submit` / `popup-comment-submit` verbatim (the manifest contract) even though Send-to-Claude is now labeled "Send to Claude" rather than "Comment" — update the visible label, keep the testid.
- **Mitigate the residual missed-send risk (plan-review MED).** The inversion is safe in the leak direction (Enter → private note, never an accidental send), but the *reverse* failure mode is real: a user with old muscle memory hits Enter expecting "send to Claude" and silently gets a private note Claude never receives. The keybinding is a locked Conflict #5 decision — but the **mitigation** must ship with it: (a) both key hints must be visibly rendered on their buttons (`⏎ Note to self`, `⌘⏎ Send to Claude`) so the mapping is discoverable, not just inverted; and (b) the post-action feedback (existing toast path) must distinguish "Saved as private note" vs "Sent to Claude" so the outcome is unambiguous.
- Anchored popover that flips above/below the selection; **no quoted-text echo** of the selected text (the mockup omits it). Largely restyling the existing annotate-mode block + the keybinding swap.
- **Tests that assert Enter=comment must flip to Enter=note** (and add a `Ctrl+Enter`=comment case). This is the test-update signal the per-spec assertions will catch — fix the behavior/labels, and update the assertions to the new contract; do **not** silence them.

### D3. Optional bar + swap control — mirror the panel-visibility idiom

- Add `formattingBarVisible: boolean` (default `true`) to settings, following the `leftPanelVisible` idiom exactly: interface field, `DEFAULTS`, `normalizeKnownFields` (`=== false ? false : DEFAULTS`), schema bump + pure migration step (v9→v10 — **note this stacks on top of 1.13's v8→v9**; if 1.13 hasn't bumped yet at implementation time, reconcile to whatever `CURRENT_SCHEMA_VERSION` is on the branch). Add nothing to `REMOVED_FIELDS`.
- Wrap the `<FormattingBar>` invocation in `{#if settingsState.settings.formattingBarVisible}` (the bar has no internal visibility gate today, so a direct `{#if}` is correct — no LayoutModel wrapper needed since there's no multi-factor override). State-loss on unmount (an in-flight link URL, an open heading menu) is real but harmless and consistent with how the codebase already gates TitleBar children with `{#if}` — the bar holds no persistence-worthy local state (unlike the editor/panels, which use display-toggle). Note it in the PR body; no display-toggle conversion needed (plan-review LOW).
- **Swap/hide affordance:** the bar gains a trailing "hide bar" control (`formatbar-hide-btn`) that sets `formattingBarVisible:false`. Because the bar is gone once hidden, **restoring** uses non-bar affordances:
  - A command-palette action `Toggle formatting bar` (register in `builtin.svelte.ts` + App.svelte handler calling `updateSettings({ formattingBarVisible: ! })`), and
  - An Appearance settings toggle (`appearance-formatting-bar`).
  - A keyboard shortcut is **optional** for 1.0 — propose `Ctrl+Alt+F` only if it doesn't collide (grep first); otherwise palette + settings suffice. Decide during implementation; don't invent a colliding binding.
- When the bar is hidden, formatting is fully reachable via the always-full popup (D1) — that's the whole point of making the popup full first.

### D4. Remove authorship from the bar (home now exists)

- Delete the `{#if onAuthorshipChange}` block (FormattingBar L89-106) and drop the `showAuthorship`/`onAuthorshipChange` props from `FormattingBar`'s `Props` (L12-17).
- App.svelte: drop those two props from the `<FormattingBar>` invocation (L1103-1108). Authorship is now wired only through `DecorationsMenu` (1.13), `Ctrl+Alt+A`, and the palette action — all of which call `updateSettings` directly, so they're unaffected.
- Remove testid `formatbar-authorship-toggle` (manifest + CLAUDE.md). Update the E2E selector that references it (`tests/e2e/toolbar-redesign.spec.ts:65` per the 1.2 plan) — point it at the new Decorations-control authorship row, or move that assertion to the Decorations spec. **Do not delete the assertion** — relocate it so authorship-toggle coverage survives.
