## Context

3. A mirrored toggle group in **Settings ‚Üí Appearance** (the canonical home; a11y annotation-pattern settings stay in the separate Accessibility tab, untouched).

**Not a flagged conflict on its own** ‚Äî it's the consolidation half of the Conflict #5 resolution. **Display-only; no `src/server/` changes; ADR-027 is untouched** (notes are still never read by Claude ‚Äî this only hides the user's *own* marks in their *own* view).

## Design ‚Äî the load-bearing decisions

### D1. Master mute is an *overlay*, not a pref-clobber

`decorationsMuted` is a single boolean that suppresses **all** decoration rendering (authorship + all three annotation types) without touching the individual prefs. **Effective visibility is computed in App.svelte**, not stored:

```
effectiveAuthorship      = !decorationsMuted && showAuthorship
effectiveComments        = !decorationsMuted && showComments
effectiveHighlights      = !decorationsMuted && showHighlights
effectiveNotes           = !decorationsMuted && showNotes
```

Restoring (un-muting) recomputes the effective values from the untouched prefs ‚Üí the exact prior set returns. No "remembered set" bookkeeping needed. This is why mute is robust: the prefs are the source of truth, mute is a transient lens over them.

**Effective values must be reactive** (plan-review MED) ‚Äî compute them either *inside each `$effect` body* (reading `settingsState.settings.*` + `decorationsMuted` there, like the existing L492/L508 effects) or via `$derived` for any value threaded to `<TitleBar>`/`<DecorationsMenu>`. A plain top-level `const` freezes at mount (`feedback_svelte_const_vs_derived`) and the menu/dispatch go stale.

**Editing a row while muted auto-unmutes ‚Äî in ONE `updateSettings` call** (plan-review HIGH). The row handler builds a single partial: `{ [field]: next, ...(decorationsMuted ? { decorationsMuted: false } : {}) }`. Two sequential calls (`updateSettings({showX})` then `updateSettings({decorationsMuted:false})`) fire the decoration + authorship effects twice and produce a transient mid-state (still-muted with the new value applied). One partial, one update.

### D2. The plugins stay dumb ‚Äî App.svelte feeds them *effective* booleans

The annotation plugin should not know about `decorationsMuted` or settings. It receives **effective** per-type visibility and filters. Two changes:

- **`AnnotationToggleMeta`** becomes a richer payload carrying effective per-type visibility (not a single `visible`):
  ```ts
  export interface AnnotationToggleMeta {
    type: "toggle-decorations";
    visible: { comment: boolean; highlight: boolean; note: boolean };
  }
  ```
- The plugin holds `let visible = { comment: true, highlight: true, note: true }` (read from localStorage at init), and `buildDecorations` gains a pre-switch filter at L56 (after the range guard, before the `switch` at L60):
  ```ts
  if (!visible[ann.type]) return;
  ```
  `apply()` updates the whole `visible` object on a `toggle-decorations` meta.

**‚öÝ The per-type filter interacts with the docChanged recovery guard ‚Äî this is the load-bearing correctness point (plan-review HIGH).** The recovery branch at annotation.ts L186-198 latches `recoveryAttempted = true` only when a rebuild returns non-empty; it exists to stop re-running O(n) `buildDecorations` on every keystroke when the Y.Map-observer fired before y-prosemirror populated the doc. Today the only way `hasAnnotations === true` yet rebuild is empty is that transient (self-heals in a keystroke). The per-type filter adds a **permanent** "annotations exist but all hidden" state (e.g. a doc of only notes, notes off) ‚Üí rebuild stays empty ‚Üí `recoveryAttempted` never latches ‚Üí **full rebuild every keystroke**, silently undoing the perf fix. **Fix:** track `hasVisibleAnnotations` (walk the map applying `visible[type]`) instead of `hasAnnotations` for the L192 recovery guard *and* the empty short-circuit, and **recompute it in both places that change visibility**: the Y.Map observer (L226) *and* the `toggle-decorations` meta branch (a type toggle changes visibility without changing the map). This single quantity subsumes the `anyVisible` short-circuit ‚Äî if nothing is both present and visible, return `DecorationSet.empty` and don't retry. Add a test asserting that with annotations present but all of a hidden type, `docChanged` does **not** trigger a rebuild (guard latched).

Authorship stays on its own plugin (`authorshipPluginKey`); the **existing** App.svelte authorship effect just changes its dispatched value from `showAuthorship` to `effectiveAuthorship` (factor in `decorationsMuted`).

### D3. localStorage stores *effective* values under one consolidated key

Replace `ANNOTATION_DECORATIONS_TOGGLE_KEY` with a single JSON key so the plugin reads one blob at init instead of three:

```ts
// src/shared/constants.ts ‚Äî replaces ANNOTATION_DECORATIONS_TOGGLE_KEY
export const DECORATION_VISIBILITY_KEY = "tandem:decorationVisibility";
// value: JSON.stringify({ comment: boolean, highlight: boolean, note: boolean })  (effective, mute folded in)
```

- `useTandemSettings.svelte.ts#updateSettings` computes effective per-type (`!muted && showX`) and writes the JSON blob, plus continues mirroring `AUTHORSHIP_TOGGLE_KEY` (now also `!muted && showAuthorship`). **Also seed both keys on initial load** (the hook's constructor / first read), not only on `updateSettings` ‚Äî today the plugin defaults to all-true if the key is absent, which is a latent #596 gap; this PR closes it so a user who has decorations muted/off doesn't see a flash of marks before the first settings write.
- The plugin parses the JSON at init with a try-catch (malformed/absent ‚Üí default all-true), mirroring the existing localStorage-guard discipline.

> **Why one JSON key, not four dedicated keys:** the key exists solely for plugin-init-before-Svelte; the live channel is the meta dispatch. One parse at init is cleaner than four `getItem`s, and authorship already has its own dedicated key so the "consistency with authorship" argument is weak. The tradeoff (tests now set a JSON blob instead of `"true"`/`"false"`) is mechanical.

### D4. Migration v8 ‚Üí v9 maps the old flag by *intent*

`showAnnotationDecorations === false` meant "the user turned all marks off" ‚Äî a **persistent preference**, not a transient mute. So map it onto the three per-type flags, leaving mute off:

```ts
// useTandemSettings.ts migration chain ‚Äî new step
if (parsed.schemaVersion === 8) {
  const allOff = parsed.showAnnotationDecorations === false;
  parsed = {
    ...parsed,
    showComments:   allOff ? false : true,
    showHighlights: allOff ? false : true,
    showNotes:      allOff ? false : true,
    decorationsMuted: false,
    schemaVersion: 9,
  };
}
```

- Bump `DEFAULTS.schemaVersion ‚Üí 9`, `CURRENT_SCHEMA_VERSION ‚Üí 9`.
- Add `showComments/showHighlights/showNotes: true` and `decorationsMuted: false` to `DEFAULTS`.
- Add four `normalizeKnownFields` coercions (`parsed.x === false ? false : DEFAULTS.x`; `decorationsMuted` uses `=== true ? true : false`).
- Add `"showAnnotationDecorations"` to `REMOVED_FIELDS` (L509) so a forward-compat blob from a future version doesn't re-surface the dead field.
- Remove the interface field `showAnnotationDecorations` (L107); add the four new fields.

Unlike the v7‚Üív8 pure-bump, this migration **does** set values because it derives three fields from one ‚Äî `normalizeKnownFields` can't infer the split.
