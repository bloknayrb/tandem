# Step 6: Auto-Update

## Summary

Add Tauri's updater plugin so the desktop app checks for new versions on launch and every 8 hours, prompts the user via native dialog, and installs + restarts on confirmation. A "Check for Updates" tray menu item provides manual control. Update manifests are served directly from GitHub Releases (already generated by the existing CI workflow).

## Motivation

The desktop app targets non-technical users who won't run `npm update -g`. Without auto-update, users stay on stale versions indefinitely. Tauri v2's updater plugin provides signed, verified updates with minimal code.

## Design

### 1. Plugin Installation

**Cargo dependency** (in the desktop-only `[target.'cfg(not(any(...)))'.dependencies]` block, alongside `single-instance` and `window-state`):

```toml
tauri-plugin-updater = "2"
```

**Plugin registration in `lib.rs`:**

Register after `tauri_plugin_process`, with a `#[cfg(desktop)]` guard matching the convention used for `single-instance` and `window-state`:

```rust
.plugin(tauri_plugin_updater::Builder::new().build())
```

**Required trait imports:**

```rust
use tauri_plugin_updater::UpdaterExt;   // for app.updater()
use tauri_plugin_process::ProcessExt;    // for app.restart()
```

**Capability in `desktop.json`:**

Add `"updater:default"` to the permissions array.

### 2. Updater Configuration (`tauri.conf.json`)

Add a `plugins.updater` block:

```json
{
  "plugins": {
    "updater": {
      "endpoints": [
        "https://github.com/bloknayrb/tandem/releases/latest/download/latest.json"
      ],
      "pubkey": "<PUBLIC_KEY_FROM_TAURI_SIGNER>"
    }
  }
}
```

- `endpoints`: Points to the `latest.json` manifest that `tauri-action` already generates and attaches to GitHub Releases via `updaterJsonPreferNsis: true`.
- `pubkey`: The Ed25519 public key used to verify update signatures. Generated once via `tauri signer generate`.

Also add to the `bundle` section:

```json
{
  "bundle": {
    "createUpdaterArtifacts": true
  }
}
```

This tells `tauri-action` to generate `.sig` signature files alongside each installer artifact. Without it, `latest.json` will reference signatures that don't exist and every update will fail verification.

### 3. Signing Keypair

**One-time local setup:**

```bash
tauri signer generate -w ~/.tauri/tandem.key
```

This produces:
- `~/.tauri/tandem.key` — private key (never committed)
- `~/.tauri/tandem.key.pub` — public key (goes in `tauri.conf.json`)

**CI secrets (GitHub repository settings):**
- `TAURI_SIGNING_PRIVATE_KEY` — contents of the private key file
- `TAURI_SIGNING_PRIVATE_KEY_PASSWORD` — the password entered during generation

### 4. Update Check Logic (Rust)

A new async function in `lib.rs`:

```rust
async fn check_for_update(app: &tauri::AppHandle, manual: bool) {
    // UpdaterExt::updater() returns Result<Updater>, not Updater directly
    let updater = match app.updater() {
        Ok(u) => u,
        Err(e) => {
            log::error!("Failed to create updater: {e}");
            if manual {
                show_update_error_dialog(app, &e.to_string());
            }
            return;
        }
    };

    let update = match updater.check().await {
        Ok(Some(update)) => update,
        Ok(None) => {
            if manual {
                // User clicked "Check for Updates" — give feedback
                show_up_to_date_dialog(app);
            }
            return;
        }
        Err(e) => {
            log::warn!("Update check failed: {e}");
            if manual {
                show_update_error_dialog(app, &e.to_string());
            }
            return;
        }
    };

    // Show confirmation dialog
    let version = update.version.clone();
    let confirmed = show_update_available_dialog(app, &version).await;
    if !confirmed {
        return;
    }

    // Download and install
    match update.download_and_install(|_, _| {}, || {}).await {
        Ok(()) => {
            log::info!("Update to v{version} installed — killing sidecar and restarting");
            // Kill sidecar BEFORE restart so the new instance can bind ports 3478/3479.
            // RunEvent::Exit fires on the *old* process, but the new process may start
            // before the old one fully exits, causing a port conflict.
            kill_sidecar(app);
            app.restart();
        }
        Err(e) => {
            log::error!("Update install failed: {e}");
            show_update_error_dialog(app, &e.to_string());
        }
    }
}
```

**Important: sidecar must be killed before restart.** `app.restart()` spawns a new Tandem process immediately. The `RunEvent::Exit` handler in the old process is not guaranteed to fire before the new process calls `start_sidecar()`. If the old sidecar is still holding ports 3478/3479, the new instance's health check will either fail or (worse) pass against the dying sidecar. Explicit `kill_sidecar()` before `app.restart()` prevents this.

**Dialog helpers:**
- `show_update_available_dialog(app, version)` — "Tandem v{version} is available. Update now?" with OK/Cancel. Returns `bool` (uses `tauri_plugin_dialog` ask dialog).
- `show_up_to_date_dialog(app)` — "You're running the latest version of Tandem." (message dialog, non-blocking).
- `show_update_error_dialog(app, msg)` — "Update check failed: {msg}" (message dialog, non-blocking).

### 5. Check Triggers

**On launch (automatic):**

Spawned from the existing `setup()` async block, after the sidecar health check passes and MCP setup completes. Non-blocking — doesn't delay app startup.

```rust
// Inside the existing tauri::async_runtime::spawn block in setup()
// ... after run_setup() ...
check_for_update(&handle, false).await;
```

**Periodic (background):**

An 8-hour interval timer spawned alongside the launch check:

```rust
let periodic_handle = handle.clone();
tauri::async_runtime::spawn(async move {
    let mut interval = tokio::time::interval(Duration::from_secs(8 * 60 * 60));
    interval.tick().await; // Skip the first immediate tick (launch check covers it)
    loop {
        interval.tick().await;
        check_for_update(&periodic_handle, false).await;
    }
});
```

**Manual (tray menu):**

Add a `MENU_UPDATE` constant alongside the existing tray constants:

```rust
const MENU_UPDATE: &str = "update";
```

Add a "Check for Updates" menu item between "Setup Claude" and the separator:

```rust
let update_i = MenuItem::with_id(app, MENU_UPDATE, "Check for Updates", true, None::<&str>)?;
```

Menu order becomes: Open Editor, Setup Claude, Check for Updates, —, About Tandem, Quit.

The menu event handler spawns `check_for_update(&handle, true)` — the `manual: true` flag ensures the user always gets feedback (either "up to date" or an error).

### 6. CI Changes

The `tauri-release.yml` workflow needs the signing secrets passed to `tauri-action`:

```yaml
- uses: tauri-apps/tauri-action@v0
  env:
    GITHUB_TOKEN: ${{ secrets.GITHUB_TOKEN }}
    TAURI_SIGNING_PRIVATE_KEY: ${{ secrets.TAURI_SIGNING_PRIVATE_KEY }}
    TAURI_SIGNING_PRIVATE_KEY_PASSWORD: ${{ secrets.TAURI_SIGNING_PRIVATE_KEY_PASSWORD }}
  with:
    tagName: v__VERSION__
    releaseName: "Tandem v__VERSION__"
    releaseBody: "See the assets to download and install Tandem."
    releaseDraft: true
    prerelease: false
    updaterJsonPreferNsis: true
    args: ${{ matrix.args }}
```

No other CI changes needed. The action already generates `latest.json` with signature data when `TAURI_SIGNING_PRIVATE_KEY` is present.

### 7. CSP Update

The webview CSP in `tauri.conf.json` doesn't need changes — the updater operates at the Rust/system level, not through the webview. No HTTP requests from the frontend are involved.

## What's Excluded

- **Tray badge / visual indicator** for pending updates — future enhancement
- **Silent background install** — always prompt the user
- **CDN/proxy** for the manifest endpoint — GitHub Releases direct is sufficient at current scale
- **Update channels** (stable/beta) — single channel only
- **Custom update UI** — native OS dialogs only
- **Download progress bar** — the `download_and_install` progress callbacks are available but not wired to UI in this step
- **Rollback mechanism** — Tauri doesn't support this natively; the previous installer remains on disk

## Error Handling

| Scenario | Behavior |
|---|---|
| Network unreachable (background check) | Log warning, retry next cycle. No user-facing error. |
| Network unreachable (manual check) | Show error dialog with message. |
| GitHub rate limited | Same as network error — transparent retry. |
| Manifest parse failure | Log error, skip. Tauri rejects malformed manifests. |
| Signature verification failure | Tauri rejects the update automatically. Logged as error. |
| Download interrupted | `download_and_install` returns error. Manual: show dialog. Background: log and retry next cycle. |
| Install failure | Show error dialog. App continues running on current version. |

## Security Notes

- **Signing chain of trust:** Ed25519 keypair generated locally, pubkey embedded in binary, private key in GitHub Actions secrets. Tauri verifies every update against the pubkey before applying. HTTPS to GitHub Releases prevents manifest tampering in transit.
- **Key rotation:** No built-in rotation mechanism. If the private key is compromised: (1) revoke the GitHub secret, (2) ship one final signed release with a new pubkey using the old key, (3) users on versions older than the rotation release require manual reinstall. Acceptable at current scale.
- **CI secret scope:** Consider restricting `TAURI_SIGNING_PRIVATE_KEY` to a GitHub Environment with required reviewers if/when the repo has external contributors with write access.
- **Windows locked binary risk:** If the old `node-sidecar-*.exe` is still running when the NSIS installer tries to replace it, the replacement may silently fail. The explicit `kill_sidecar()` before `app.restart()` should prevent this, but the upgrade path with a running sidecar must be tested on Windows.
