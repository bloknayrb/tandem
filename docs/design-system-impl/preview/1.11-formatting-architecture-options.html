Sub-PR 1.11 · formatting architecture · decision aid (scratch — not committed)

Where does formatting live?

Two configurations. Option 1 keeps the persistent formatting bar and gives the selection popup a minimal role. Option 2 follows the bundle's actual A8 contract — no permanent format bar; the selection popup is the only formatting surface — so every formatting control collapses into the popup. Option 2 is mocked four ways below, with a recommendation.

Everything is interactive — chevrons, segments, overflow menus, Annotate. Switch themes to check contrast.

Option 2 Popup is the only format surface

No persistent formatting bar. Every formatting control moves into the selection popup. The chrome gets much calmer; the popup gets denser. Four designs follow — the tradeoff between them is how much is shown at rest vs. on demand.

The catch — homeless controls. A8 only accounts for selection-scoped formatting. Three controls on today's bar are not selection-scoped: Undo, Redo, and the authorship toggle. A selection-only popup can't host them. Option 2 only works if those move to the title bar (undo/redo as a small left cluster; authorship moves back where it lived pre-1.2) — otherwise undo/redo become keyboard-only, a regression for mouse users.

Bundle-faithful

The bundle's literal A8 layout, plus the audience action. Everything visible, no hidden state. Heaviest at rest — but honest: the only format surface shows all its tools.

Design 2b — inline marks at rest, block-level on demand

Recommended

Collapsed shows the high-frequency inline marks (B / I / S / code / link), highlight, and Annotate — the chevron reveals block-level controls (heading, lists, quote, rule). Calm common case, full set one click away. Two-click cost only hits the rare block actions, never bold/italic/highlight/comment.

Design 2c — segmented: Format / Annotate

Alternative

A segment at the top switches the popup between a full Format view and the Annotate note card. Clean separation, scales to any number of controls — but adds a mode switch, and formatting + commenting are never co-visible.

Alternative

High-frequency controls stay inline; the long tail (lists, quote, rule, code block) collapses behind a "⋯" menu. Narrowest silhouette, but hides discoverable controls behind a click.

Scope — Option 2 is not one sub-PR. Option 1 fits inside 1.11 as planned. Option 2 is a 2–3 PR sequence: build the consolidated popup → relocate undo/redo + authorship to the title bar → delete the persistent FormattingBar → retire/rename its testids (formatting-bar, formatbar-authorship-toggle, toolbar-highlight-*, toolbar-link-*) → re-point tutorial anchors → reopen Conflict #2 with an override (1.2 just visually ported the bar that 2 deletes). Bigger payoff, materially bigger job.

Recommendation

If you go Option 2: lean Design 2b, with 2a a genuinely close runner-up — and relocate undo/redo + authorship to the title bar either way.

  • 2b vs 2a is close, not obvious. The popup is already context-gated (it only appears on selection), so 2a's "everything one click, pill is just taller" is a real, predictable option — the bundle author chose it deliberately. 2b's edge is purely a frequency bet: most selections end in highlight or comment, few in blockquote/rule, so paying a second click only on the rare block actions feels right. If you value predictable muscle-memory over a calm resting state, 2a wins.
  • One wrinkle that favors 2a: the H⌄ heading dropdown is itself a popover. In 2b it nests inside the chevron-expanded row inside the selection popup — popover-in-popover-in-popup, with fiddly escape / click-outside / focus-return semantics. 2a (and 2c) keep H⌄ at the top level of an always-visible row, one less level of nesting.
  • 2b/2a over 2c: a segment makes formatting and commenting both cost a mode switch. Audience-first means the comment path should be a single visible button.
  • 2b/2a over 2d: overflow menus hide discoverable controls; a labeled expander or a visible second row reads as "more here" better than "⋯".
  • Option 1 is the safe play: lowest risk, no homeless-control problem, smallest diff, fits 1.11 as-is — but it leaves two formatting surfaces standing.
