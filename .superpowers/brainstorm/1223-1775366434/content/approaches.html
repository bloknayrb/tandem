Three Approaches to .docx Batch Apply

All three use XML-level replacement via JSZip. They differ in how they map Tandem offsets → document.xml positions.

A

Parallel Walker (Recommended)

How: Reuse the offset-tracking logic from docx-comments.ts to build a reverse map: flat offset → (paragraph index, run index, char index within <w:t>). Walk document.xml once, record positions for every accepted suggestion's range, then apply replacements in reverse order.

Pros: Proven pattern (comment extraction already does this walk). Single-pass offset calculation. Handles heading prefixes, tabs, breaks correctly because it shares the same walker. Smallest blast radius — only touches text in <w:t> elements.

Cons: Two-pass (walk to map, then apply). Must handle suggestions that span multiple runs (split/merge <w:r> elements). The walker logic needs to be extracted from docx-comments.ts into a shared module.

Complexity: Medium. ~300-400 lines of new code + refactoring the shared walker.

B

Text Stream Splice

How: Serialize all <w:t> content in document.xml into a flat string (preserving XML node references), apply string-level replacements at the flat offsets, then distribute the modified text back across the original <w:t> elements.

Pros: Conceptually simple — work at the string level, then project back. Handles cross-run replacements naturally since you're operating on the flat string.

Cons: Fragile reconstruction — splitting modified text back across <w:r> elements with their original formatting is error-prone. Formatting at splice boundaries may be wrong (bold/italic runs get corrupted). Harder to generate proper tracked-change markup.

Complexity: Medium-high. Simpler logic but harder edge cases.

C

OOXML Library (docx-merger or python-docx via subprocess)

How: Use a dedicated OOXML manipulation library that understands Word's object model. Apply changes at the semantic level (paragraphs, runs, text) rather than raw XML.

Pros: Library handles run splitting, formatting preservation, and tracked changes natively. Less custom XML manipulation.

Cons: No good Node.js OOXML library exists for in-place editing (the docx npm package is generation-only, not editing). Would need python-docx via subprocess (heavy dependency, deployment complexity) or a less-maintained library. Offset mapping still needed — libraries work with paragraph/run indices, not flat offsets.

Complexity: Low code, high dependency risk. Still need the offset mapper regardless.

Recommendation: Approach A

The parallel walker builds on proven code (docx-comments.ts), keeps dependencies minimal (just JSZip, already installed), and gives us precise control over tracked-change markup and comment resolution. The refactoring of the shared walker is work we'd want to do anyway — it eliminates duplication between read and write paths.

The main risk — cross-run suggestion replacement — is bounded: we can handle it by deleting spanned runs and inserting new <w:r> elements with the replacement text, inheriting formatting from the first run in the range.
