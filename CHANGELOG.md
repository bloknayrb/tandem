## [Unreleased]

### Added

- **Raw-markdown source view / edit mode (#1021)** — Tandem's editor is WYSIWYG, but documents persist as raw markdown on disk; there was no way to see or edit that underlying source in the app. A new **`</>` toggle in the title bar** (and a **"Toggle markdown source view"** command-palette action) swaps the active `.md` document between the formatted editor and a full-document **editable** markdown textarea. Editing the source and returning to WYSIWYG round-trips the text through the same `remark` parser the autosave uses, then persists to disk; the source you see is byte-identical to what's written on disk (`saveMarkdown`). The toggle only appears for **editable `.md`** documents — non-`.md` formats and read-only `.md` (e.g. `CHANGELOG.md`, `.docx`) are excluded, both in the UI affordance and server-side. Because a source edit re-anchors the whole document, **annotations are cleared on round-trip** (the same trade-off as `tandem_open force:true`); an inline notice warns before you commit when annotations exist, and entering source view with *no* edits restores the formatted view untouched. Two new loopback routes back this: **`GET /api/document/raw`** (returns the literal markdown) and **`POST /api/document/reload`** (replaces Y.Doc content from a supplied markdown string; 1 MB cap on the inline body since the parse is synchronous, plus open/`.md`/not-read-only/not-already-reloading guards). The reload reuses the existing force-reload lifecycle (`clearAndReload`) but keeps the doc dirty so the user's edits are written, and serializes against the file-watcher reload via the shared in-progress guard. `Ctrl+S` in source view commits the edit instead of saving stale Y.Doc content. New `titlebar-source-toggle` + `source-view-*` testids; no annotation-schema or coordinate-system change. Plan reviewed pre-code by adversarial + security + editor-null-audit agents (which caught the annotation-store-wipe, read-only, dirty-flag, and concurrency-guard requirements before any code).
- **Settings → Appearance: "Match System" can resolve light to Warm (#993)** — when Theme is set to **System**, a new "Use Warm when system is light" toggle makes a light OS appearance resolve to the paper-tone **Warm** theme instead of neutral **Light**, while a dark OS appearance still resolves to **Dark**. Previously System could only ever pick Light or Dark, so Warm fans had to give up OS-driven dark-mode tracking to keep the warm canvas. The preference (`systemLightVariant`, default `"light"`, schema v13→v14) is threaded through `systemTheme()`/`resolveTheme()`/`applyTheme()` so both the initial resolve and the live `prefers-color-scheme` re-resolve honor it. New `appearance-system-light-warm` testid (shown only while Theme is System); no behavior change for existing users.
- **Rename an open file from the UI (#1017)** — there was previously no way to rename a file once it was open; you had to rename on disk (orphaning the tab's annotations and forcing a reload) or close and reopen. Now **double-click a tab's name, or press F2**, to edit the filename in place — Enter commits, Escape cancels. The rename keeps the same directory and extension (no format conversion, no move); scratchpads, uploads, and read-only docs (incl. `.docx`) are not renamable (they use Save As). Architecturally the **documentId / Hocuspocus room stays stable** — only the on-disk path migrates, exactly like Save-As's promote-in-place — so connected clients keep their live CRDT room and just see a new tab label; **annotations follow the file** (the durable envelope is migrated from the old path-hash to the new one, with a heal-write that closes a cross-doc recovery-steal window). A new **`POST /api/rename`** route and **`tandem_rename`** MCP tool expose the same operation to agents (MCP-first, ADR-038). Free-form names are validated server-side against path separators, `..`, Windows-illegal characters (notably `:`, the NTFS alternate-data-stream vector), reserved device names (`CON`/`NUL`/`COM1`…, keyed on the first-dot stem), trailing dots/spaces, UNC/`\\?\` prefixes, and symlinked targets; the extension must match case-insensitively. As part of this PR a pre-existing inverted flush order in `closeDocumentById` was fixed (Finding C) — a delete-then-close within the annotation debounce window could drop the tombstone and resurrect a deleted annotation; both the close and the new rename path now flush the durable store *before* tearing down the file-sync context. New `tab-rename-input-{id}` / `tab-name-{id}` testids; no annotation-schema or coordinate-system change.
- **Settings → Editor: default save folder (#1023)** — a new "Default Save Folder" preference controls where new files land when you use **Save As** in the desktop app. Pick a folder with the native picker (or type a path), and the save dialog opens there. When unset, Save As falls back to a smart default: your AI's configured working directory, then your home folder. The setting is stored client-side (independent of the Claude working directory) and is a no-op in the browser distribution, where Save As is a download. The integration lookup is time-boxed so a slow/absent server never blocks the save dialog. The native **Open** dialog now starts in this same smart-default directory too (previously it always opened at the OS default), so "where my files live" is one consistent notion across open and save; the three-tier resolution is factored into a shared `utils/default-directory.ts` consumed by both dialogs.

### Changed

- **Updater audit — no rollback / no in-updater post-restart health-poll for v1, documented (#925)** — audited the Tauri auto-updater (`perform_install`) against Tauri v2 best practices and CrabNebula's auto-updates guide for the two gaps #925 raised: missing rollback on a failed relaunch, and a missing post-restart health-poll. The decisive constraint is that Tauri v2 `AppHandle::restart()` is **divergent** (exits + relaunches without returning), so an in-`perform_install` post-restart probe is unreachable dead code and rollback can't be driven from the old (now-gone) process. The post-restart health verification the issue wants **already exists for the sidecar** (`start_sidecar` → `wait_for_health` → `MAX_RESTARTS`, native "Server Error" dialog on exhausted failure; the `sidecar-restart-failed` WebView event is manual-`restart_sidecar`-only); only the rare Tauri-*shell* relaunch failure is uncovered, which neither cited reference handles (both stop at `download_and_install()` + relaunch). **Decision (c): ship v1 unchanged**, with the one in-process-only hardening that survives the constraint — a persisted "pending update" marker the next boot reads to surface a "your update may not have completed" hint — sketched as an optional, build-verified follow-up. Documented in `docs/spikes/updater-rollback-healthpoll-audit.md` and ADR-043 (`docs/decisions.md`). Docs-only; no code or behavior change.
- **Stop hook injects its workflow reminder as real context (#1045)** — `.claude/hooks/stop-cycle-check.sh` now emits its "uncommitted source edits" reminder on STDOUT as a `hookSpecificOutput.additionalContext` JSON object (built with `jq`, falling back to stderr when `jq` is unavailable), so Claude Code surfaces it as actual context instead of stderr-only noise. The one-shot-per-session gate (source-edit marker present, not already nudged, no newer commit) and `exit 0` semantics are unchanged; the no-pending-edits path still emits nothing. The version-pinning sub-question (min/max Claude Code version via managed settings) is deferred as a separate design decision.
- **Softer, friendlier wording on the two flagship error surfaces (#1016)** — the app-root error boundary and the lost-connection banner read less blunt without losing accuracy. The error boundary's heading goes from "Something went wrong" to "Something didn't go as planned", its body from "The editor encountered an unexpected error." to "The editor ran into an unexpected hiccup.", and the exhausted-recovery line from "Recovery attempts exhausted. Reload the page to continue." to "We weren't able to recover this time — reloading the page should get you going again." The connection banner goes from "Connection to the Tandem server has been lost. Ensure the server is running." to "We've lost the connection to the Tandem server. Please check that it's still running." Copy-only — no error codes, control flow, or testids changed; the still-accurate "reload" / "check the server is running" guidance is preserved, and no `err.message` or path is leaked. Unit + E2E assertions on the old strings were updated to match.
- **Clarify what "Decorations" means in the title-bar control (#1020)** — the title-bar Decorations dropdown now leads with a one-line explanation that decorations are the inline editor overlays (author colors, comment/highlight/note marks), and that toggling a type hides its marks in the document while side-panel cards stay visible. This matches the help text already shown under Settings → Appearance → Decorations and resolves the confusion that the setting did nothing visible (it was previously labeled only "Decorations" with no description). Defaults are unchanged — all four decoration types remain on by default — and no testids changed.
- **New-tab morph — clear two-phase motion (A29, follow-up to #977)** — the `+`→menu morph now expands **horizontally into a clean capsule first, then vertically into the card**, instead of co-timing both axes. The radius is held at a literal capsule value (`14px`, half the pill height) through the horizontal unroll and squares to the card radius only as the height grows, so the shape never flashes a flat lozenge; the interior rows cascade in during the vertical phase. Close reverses it. The card fill persists through the whole collapse (a *filled* card shrinks back into the `+`, with no empty-box gap), and the card/pill shadows crossfade so there's no shadow pop at the handoff. On close, focus is **restored to the element that was focused before the menu opened** (typically the editor) rather than parked on the `+`, so no stray focus ring lingers on the pill; the closing body is `inert` so focus is never stranded in the clipped panel, and on close the in-menu search field is **blurred synchronously** so a keyboard shortcut fired in the same tick (e.g. Escape then `Ctrl+W`) is no longer swallowed by the still-focused input. Fully reduced-motion-correct (all timing derives from the shared morph tokens).
- **A discoverable "Connect AI" entry point when AI isn't running (#1018/#1022)** — the app now surfaces a CTA when AI isn't actually working, instead of leaving users "Synced" with no way to find setup. Readiness keys on the launcher's `GET /api/launcher/status` truth — not the document-sync dot, and not `claudeActive` (which flaps to idle every few seconds and would make a working user's chip oscillate). When Claude Code isn't set up, a **"Connect AI"** chip appears (titlebar, secondary) and a primary CTA shows in the empty state, both opening the Claude Code wizard; when it's configured but stopped/crashed, the CTA becomes **"Restart Claude Code"** and relaunches it (reusing the existing relaunch command) rather than wrongly sending an already-set-up user back into setup. The prompt is **suppressed in Solo mode** (where working without AI is intentional) and never flashes during cold boot (gated on first-run-settled + connected + a successful status read; a transient status-fetch failure keeps the prior state). New `useAiReadiness` hook + `titlebar-connect-ai` / `empty-state-connect-ai` testids.
- **Loud-failure notices when you message AI but none is connected (#1018)** — sending a chat message or a "Send to Claude" comment while no AI is connected now pops a toast that **affirms the save and frames absence as deferred delivery** ("Message/Comment saved — no AI is connected yet. It'll be seen when AI connects"), with an inline **"Connect AI"** / **"Restart Claude Code"** action button — never "failed" or "lost" (the message/comment persists in the Y.Doc and is read whenever an agent next connects). The notice fires **only for outbound comments and chat**, never for notes or highlights, which are user-private and never sent to AI (ADR-027) — a "no AI connected" notice on those would be misleading. Suppressed in Solo mode and while readiness is still booting. Toasts gained optional action buttons (client-only — the handler never touches the SSE wire or the persisted activity tray). New `toast-action-{id}` testid; `data-ai-chip` state attribute added to the readiness chips/CTAs for test disambiguation.
- **First-run leads with Claude Code setup; in-app "bring your own API key" Models UI hidden until it works (#1018/#1022)** — AI in Tandem today is the external Claude Code integration (MCP). The in-app Models registry stores provider keys in the OS keychain, but no server-side LLM client consumes them yet, so configuring a model there did nothing — users entered a key, AI still didn't work, and concluded the app was broken. The entire BYO-models surface (first-run model picker, Settings → Models tab, titlebar default-model chip) is now gated behind a `BYO_MODELS_ENABLED` flag (off) until the outbound LLM client lands; with it off, first-run leads straight to the Claude Code wizard. The keychain storage layer and its server tests are untouched, so the path re-enables cleanly when wired.
- **Titlebar connection label "Connected" → "Synced" (#1018)** — the green status dot reflects the document sync server, not whether AI is reachable. The literal word "Connected" implied AI was connected, which was the core of the "connected but no AI" confusion. The dot's color and behavior are unchanged; only the label is clearer.

### Fixed

- **Windows: the window keeps its rounded corners when snapped and no longer shows an outline around the app shell (#984)** — the borderless (`decorations: false`) window relied on default Windows 11 behavior, which squares the corners on snap/maximize and could draw a thin 1px border. The desktop shell now opts in to `DWMWA_WINDOW_CORNER_PREFERENCE = DWMWCP_ROUND` and sets `DWMWA_BORDER_COLOR = DWMWA_COLOR_NONE` via `DwmSetWindowAttribute`, applied at window setup and re-asserted on `Resized` (snap/maximize always resize the window and reset the corner preference). Windows-only behind `#[cfg(target_os = "windows")]`; a no-op on macOS/Linux. A pre-Win11 `DwmSetWindowAttribute` failure is silently ignored (best-effort) and never aborts startup. (Requires manual verification on Windows 11.)
- **Rename no longer drops a durable annotation that is newer in the file than in the live doc (#1040 × #1041)** — a data-loss regression surfaced by the interaction of two already-merged PRs. #1051's rename now writes the new path-hash envelope from a **read-modify-write of the live Y.Doc** (rather than moving the old envelope), but `snapshot()` reads *only* the live Y.Maps. When a durable annotation was **newer in the OLD file envelope than in the live doc** — e.g. a note flushed at `rev 2`, then diverged to `rev 1` in the live map by a `withInternal` write the durable-sync observer skips (so no debounced write re-converges the envelope) — the pure-live RMW snapshot clobbered the new envelope with the stale `rev 1` record, and the subsequent re-wire's `loadAndMerge` (now reading the clobbered envelope) found nothing newer than live, so the `rev 2` file record was **lost on rename**. In the common case (live == file) this was invisible; #1041's strengthened note-privacy test exercised the divergent case and exposed it. The rename now **folds the old envelope's alive records forward into the live doc before the RMW snapshot** (new `mergeEnvelopeForward` helper reusing the existing `mergeMap`/`pickWinner` file-wins-by-rev path), so the file-newer record wins, lands in the RMW snapshot, and survives under the new path-hash. The fold writes through `withFileSync` (no raw `doc.transact`, no channel/durable echo, ADR-027-safe for notes), respects the effective tombstone ledger so a just-deleted record isn't resurrected, and is best-effort wrapped so it can never flip a committed rename to "error" (worst case degrades to the prior live-only snapshot). All of #1051's resurrection/rollback/step-2 fixes are preserved. A new regression test asserts a comment that is newer in the OLD file wins the rename merge and survives durably (verified red before the fix); the #1041 note test remains the primary ADR-027 guard. No annotation-schema or coordinate-system change.
- **Editor links now show a pointer cursor and reveal their destination on hover (#996)** — anchors in the editor previously rendered with the default text cursor and gave no hint of where they pointed. They now carry `cursor: pointer` (scoped to `.tandem-editor a[href]`) and a native `title` tooltip mirroring the link's `href`, so hovering surfaces the destination URL. The `title` is injected by extending the Tiptap Link mark's `renderHTML` (a `.docx`-imported explicit title still wins); click handling and the `openOnClick: false` navigation intercept are unchanged.
- **Rename no longer has sub-second windows that resurrect a just-deleted annotation (#1040)** — closing the concurrency windows documented (honestly, as "duplicate/resurrect, never lose") when #1017/#1038 landed open-file rename. The envelope migration now collapses the old `fs.rename` + heal-write split into a single **read-modify-write**: the new path-hash envelope is written *with the corrected `meta.filePath` already in place* **before** the old envelope is removed, so there is no instant where a byte-identical concurrent open can match a stale envelope and steal it (windows b and c). For the observer-detached gap (window a), the old-hash annotation observer now stays attached across the rename instead of being torn down up front, so a DELETE arriving mid-rename still records a tombstone, and that tombstone ledger is migrated forward into the new path-hash envelope so the deletion survives a later stale-tab merge. A follow-up review (`crdt-reviewer` + `annotation-model-reviewer`) found a **residual variant of window a**: a DELETE arriving during the post-rename flush IO, or during the re-wire's `loadAndMerge` envelope read, was recorded into the *old*-hash ledger by the still-attached observer with no later fold to carry it forward — and `loadAndMerge`'s seed *clobbered* the new-hash ledger from the just-written file (which didn't yet carry that tombstone), so the deletion could be discarded and the annotation re-inserted from the file. This is now closed on two complementary fronts: (1) `loadAndMerge` **unions** the file-derived tombstone seed with any pre-existing in-memory ledger (highest rev per id) instead of overwriting it — and applies the unioned set, so a migrated-forward delete is never dropped; and (2) the old→new ledger fold is performed inside `loadAndMerge` **after its `store.load()` read but before the merge**, *gated to the genuine rename* (an explicit `migrateTombstonesFrom` oldHash threaded through `wireAnnotationStore`), so a DELETE arriving during the read is folded forward and applied rather than resurrected — never folding tombstones across unrelated open/close. A final synchronous flush after the re-wire guarantees the migrated tombstone reaches disk before the rename returns. The force-reload path is unaffected (it clears the ledger and the envelope before re-wiring, so the union starts empty). The load-bearing `closeStore`-before-teardown ordering (#1017 Finding C) is preserved. All Y.Doc writes in the heal path go through `withFileSync` (no raw `doc.transact`); the tombstone fold is pure in-memory ledger mutation. New regression tests inject a DELETE during the RMW envelope write and during the `loadAndMerge` read and assert it stays deleted through reopen (each verified red before the fix); A second review caught a **HIGH regression on the failed-rename rollback path**: keeping the old observer attached (the window-a success-path fix) meant that on a rolled-back rename `oldHash` still matched the registered context, so the rollback re-wire's `loadAndMerge` re-seeded the old-hash tombstone ledger and the trailing `setFileSyncContext` then `"close"`-disposed the still-present same-hash context — deleting the ledger it had just repopulated, after which a stale-tab snapshot could overwrite `<oldHash>.json` with an empty tombstone list and resurrect the annotation. Fixed by restoring the master ordering **on the rollback path only** — `clearFileSyncContext(docId)` is called *before* the rollback re-wire so there is no stale same-hash context left to `"close"`-dispose after the re-seed (safe because Phase 1's `closeStore` already flushed the ledger to disk and nothing was renamed, so there is no concurrent-delete window on rollback); a new rollback regression test asserts the tombstone survives the rollback + a stale-merge snapshot + reopen (verified red before the fix). The forward `migrateTombstoneLedger` fold now spreads the source record so `.passthrough()` tombstone fields carry over. A final hardening pass closed a residual on the rename **success** path: the Phase-3 re-wire at the new path is wrapped in a swallowing try/catch, so if `wireAnnotationStore(... newPath ...)` throws internally, `setFileSyncContext` never runs and the **old**-hash observer stays registered and live (now pointing at the vanished `oldPath`) — after which RMW step-2's `clear()` of `<oldHash>.json` no longer protects against a concurrent DELETE that schedules a *fresh* debounced write **after** the clear, re-creating `<oldHash>.json` with the vanished path (a stale-envelope steal vector). The re-wire now records a `rewired` boolean and, **only when it failed** (`!rewired`), disposes the stale old-hash observer via `clearFileSyncContext(docId)` before the step-2 clear — gated on `!rewired` because on the success path `setFileSyncContext` already disposed the old context and `docId` points at the new hash, so an unconditional clear would tear down the freshly-wired new-hash observer. A new regression test forces the re-wire to throw, fires a DELETE after the rename returns, and asserts `<oldHash>.json` is not re-created (verified red without the guard); added `loadAndMerge` unit tests cover the union-not-clobber `seedHasExtraTombstones` branch and the cleared-ledger force-reload safety. reviewed pre-merge by the `crdt-reviewer` and `annotation-model-reviewer` agents. No annotation-schema or coordinate-system change.
- **New-tab menu opens rightward instead of off-screen when the `+` is near the left edge** — the A29 morph menu is right-anchored and grows left into its 460px card, but the title-bar tab strip left-justifies against the brand cluster, so with few tabs the `+` sits near the left viewport edge and the menu spilled off-screen to the left. The menu now measures the `+` slot on open and flips to grow **rightward** when there isn't room to grow left (re-anchoring both the shell and the `+` so the pill stays pinned at the slot). It stays direction-symmetric — with many tabs the `+` is far enough right that the menu keeps growing left toward the tabs.
- **A scratchpad saved to disk now reliably appears in the New Tab "recent files" list (#1019)** — when you promote a scratchpad to a real file via **Save As…** (e.g. `Scratchpad.md`), the new on-disk path is now registered in recents the instant the server confirms the write, instead of relying solely on the asynchronous openDocuments-broadcast → tab-reconcile → recents-sync effect chain (which could miss the file if the tab was closed before the round-trip completed). `runTauriSaveAs` records the server's resolved `targetPath` via the same `addRecentFile` path the file-open dialog and launcher already use, so it dedups cleanly against the later effect-driven add (no duplicate entry). Browser-mode Save As is unchanged: it downloads a blob without promoting the doc, so the still-ephemeral `upload://` scratchpad is correctly kept out of recents. localStorage access stays guarded (the recents helpers already wrap reads/writes in try/catch).
- **Selection popup no longer covers the highlight-color sub-menu (#1024)** — opening the highlight-color picker from the formatting bar while text is selected previously rendered the color swatches *behind* the selection pill. The formatting bar is a `position: fixed` wrapper that creates its own stacking context at `--tandem-z-sticky`, which sits below the selection popup at `--tandem-z-modal` — so any dropdown opened from the bar was trapped beneath the pill. The bar's wrapper now lifts to `--tandem-z-toast` (above the selection popup) only while the highlight sub-menu is open, so the sub-menu paints fully clear of the pill and drops back to `--tandem-z-sticky` on close. The sub-menu stays anchored to its toggle (z-index only; no positioning change). Visual-only; every testid preserved (`toolbar-highlight-color-toggle`, `toolbar-highlight-color-{yellow|green|blue|pink}`).
- **Onboarding tutorial copy now matches the UI control label (#1015)** — the in-app tutorial's "Ask a question" step told users to "click Comment", but the selection toolbar control is labeled **Annotate** (clicking it opens a popover whose buttons are "Send to {AI}" and "Note to self"). The step now reads "click Annotate, then send it to your AI assistant — or keep it as a private note to yourself." The matching `docs/user-guide.md` walkthrough is corrected the same way (it also dropped a stale "toggle @Claude" instruction that no longer exists in the UI). Copy-only; no behavior change, every testid preserved.
- **`tests/server/platform.test.ts` no longer binds hardcoded ports in the Windows reserved range (#1014)** — the three `waitForPort` tests bound fixed ports `49170`/`49171`/`49172`, which fall inside the IANA dynamic/ephemeral range that Windows reserves in chunks (Hyper-V/WSL/Docker). On such hosts those binds throw `EACCES` deterministically, failing the husky pre-push gate and forcing a blanket `HUSKY=0` bypass. The tests now bind an OS-assigned port (`listen(0)`) and use the resolved port, eliminating the reserved-range collision. Test-only; no production impact.
- **Closed a test-coverage gap in `resolveWordComments` (#1012)** — the suite exercised the function only with a single `importCommentId` per call. Added two tests covering the multi-comment path: resolving two distinct comments in one call (each mapped to its *own* last-paragraph paraId) and the dedup-by-`commentId` filter (two suggestions sharing one comment, asserting a single `commentEx` entry). Test-only; the production path was already correct.

### Tests

- **Strengthened the ADR-027 note-privacy rename test to cover the live-note `withFileSync` merge path (#1041)** — the existing rename note-privacy test got zero channel events "for free" because `closeStore` re-converges the moved envelope to equal the live Y.Map, so `loadAndMerge`'s `withFileSync` block performed no mutation. The new variant defeats that convergence (flush a note at `rev 2`, then diverge the live Y.Map to `rev 1` via a `withInternal` write the durable-sync observer skips), so `loadAndMerge` actually runs `annMap.set` on a live note inside the `withFileSync` transaction during rename — then asserts the note still leaks neither via channel events nor via the `getDocumentStore` read path `tandem_getAnnotations` uses. Test-only; no `src/server/` change.

## [0.13.6] - 2026-06-04

### Added

- **Native context menu on annotation cards (#923, Phase 3)** — right-clicking an annotation card — in the right rail *and* the margin view (including collapsed stub pips) — now opens a native OS menu instead of the bare WebView default. It's a faster path to the actions that already exist as card buttons, grouped with collapsing separators: **Accept** / **Dismiss** (non-user, pending), **Reply…** (note + comment — opens the in-card composer), **Edit…** (opens the in-card editor), **Send to Claude** (user notes), **Copy text** (the annotation body → clipboard), and **Archive** (note) / **Remove** (highlight / comment). Built on the Phase 1/2 plumbing with zero new IPC machinery: the request crossing the Tauri boundary is **booleans only** — the annotation id never leaves the webview (it's captured in a module-level gesture singleton) — and Rust emits one of a closed `ctx:annotation:*` id set back through the same shared `forward_context_menu_event`. The emitted id is **re-validated against the live annotation** before dispatch (the modal popup can be held open across a status change), and the gating predicates are **extracted and shared with the in-card buttons** so the menu and the buttons can't drift. As part of this, the **Edit affordance is corrected to user-authored annotations only** (`author === "user"`) on both the menu and the existing button — a user shouldn't rewrite Claude's or an imported comment's text (`tandem_editAnnotation` is Claude's own path); the spurious Reply button on highlights is likewise gone. Because the rail and margin can be open simultaneously, the `context-menu-action` listener is **shared and refcounted** (idempotent teardown so a panel unmount can't dead-key the survivor). Browser (npm-install) mode keeps its native WebView menu. ADR-027 is unchanged — the note-privacy boundary is server/MCP/channel, so replying to your own note (or copying its text to your own clipboard) is fine. No `src/server/` changes, no new testids (native menu); a new `data-annotation-id` attribute on the card root is the delegation hook. New: `src/client/panels/annotation-context-menu.ts` (pure predicates + request builder) and `annotation-context-menu-host.ts` (gesture singleton + refcounted listener + shared `runAnnotationAction`); `show_annotation_context_menu` + `build_annotation_context_menu_spec` in `src-tauri/src/lib.rs`. Plan reviewed pre-code by four adversarial agents (svelte-migration, annotation-model, crdt, security) — which caught the refcount-underflow dead-menu, the snapshot-vs-live-state dispatch, the import-note gating, and the gateless-remove re-validation before any code. The native popup needs a manual desktop pass (not E2E-automatable).
- **Imported Word reply threads & private note replies (#1000)** — Word reviewer comment *threads* now import as real reply threads: the root comment becomes a note (as before) and its replies attach to that note as a reply thread, instead of being flattened into separate stray notes or dropped. More broadly, notes can now carry reply threads at all — you can reply to your own notes, and imported Word replies show the original reviewer's name as a byline. **These note/import replies are user-private and never sent to Claude**, mirroring how note bodies are already private (ADR-027): privacy is stamped on each reply at creation (`AnnotationReply.private`), so promoting a note to a comment ("Send to Claude") surfaces the note body but keeps the pre-promotion thread local — it is never back-published. - **Authorship gutter no longer renders as a full-height bar down the editor** — the per-paragraph authorship thread (#518B) anchored its `position: absolute` `::before` to the wrong element: the intended positioning-context rule used a `.tandem-editor .ProseMirror p` descendant selector, but Tiptap stacks both classes on a single node, so the selector matched nothing and non-empty paragraphs stayed `position: static`. ## [0.7.0] - 2026-04-20

### Added

- **Auth token storage** — on first boot the server generates a 32-byte base64url token and persists it to the platform data directory (`%LOCALAPPDATA%\tandem\Data\auth-token` on Windows, `~/.local/share/tandem/auth-token` on Linux, `~/Library/Application Support/tandem/auth-token` on macOS). Token comparison uses SHA-256 on both sides before `crypto.timingSafeEqual` to eliminate the length oracle. Rate-limiting (5 attempts / 60 s) keyed by IPv4 address or IPv6 `/64` prefix with LRU eviction; Authorization headers are redacted from all rejection logs.
- **TANDEM_BIND_HOST**\*\* bind-mode selection\*\* — MCP HTTP server binds to `127.0.0.1` by default; set `TANDEM_BIND_HOST=0.0.0.0` (or a specific LAN IP) to expose Tandem on the local network. Hocuspocus WebSocket always stays loopback. Non-loopback bind without a token file exits 1 with guidance; `TANDEM_ALLOW_UNAUTHENTICATED_LAN=1` is the escape hatch. Multi-homed machines require `TANDEM_LAN_IP` to be set explicitly.
- **tandem rotate-token** — new CLI subcommand that atomically regenerates the auth token, notifies the running server to open a 60-second grace window for in-flight sessions, and re-runs `tandem setup` across all detected MCP config files. Prints old and new token fingerprints (first 8 hex chars of SHA-256). Refuses rotation when `TANDEM_AUTH_TOKEN` is set in the environment (Tauri mode).
- **Token forwarding in stdio bridge, monitor, and channel sidecars** — `tandem mcp-stdio`, `tandem monitor`, and the channel sidecar now forward `TANDEM_AUTH_TOKEN` as `Authorization: Bearer` on upstream HTTP calls. Malformed tokens (empty, `Bearer`-prefixed, < 32 chars, non-URL-safe) exit 1 with a specific message.
- **OAuth protected-resource metadata** — `/.well-known/oauth-protected-resource/mcp` now declares `bearer_methods_supported: ["header"]` and a literal-`localhost` `resource` field per RFC 9728.
- **/health**\*\* session-presence guard\*\* — `hasSession` is omitted from `/health` responses on non-loopback requests, preventing session-presence leakage on LAN binds.

### Security

- Loopback detection keys off `req.socket.remoteAddress` exclusively — `Host` header is never trusted for the loopback bypass decision.
- Fail-closed on LAN bind: `TANDEM_BIND_HOST=0.0.0.0` without a token file exits 1; the server never auto-generates a token and proceeds silently.
- `crypto.randomBytes` failure or non-writable data directory → server exits 1; no silent fallback.
